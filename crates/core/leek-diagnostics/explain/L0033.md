# L0033: chained comparison

A comparison is chained like math notation: `a < b < c`. In
LeekScript this does **not** test "b between a and c" — comparisons
are left-associative, so it parses as `(a < b) < c`: a boolean
compared against `c`. Since booleans coerce to 0 or 1, `0 < x < 10`
is true for *every* x, and the bug hides silently.

Erroneous example:

```leekscript
if (0 < x < 10) {     // really (0 < x) < 10 — always true
    moveToward(getNearestEnemy())
}
```

Write the two comparisons out with `&&`:

```leekscript
if (0 < x && x < 10) {
    moveToward(getNearestEnemy())
}
```

This is a classic trap when coming from math notation or Python,
where chaining works as written. The lint is in the default-on
*suspicious* group because the chained form is almost never what was
meant. Silence a deliberate use with `// @allow(L0033)`.
