# W0511: migration behavior change

`miku migrate` found a construct that stays syntactically valid
across the version boundary but *means something different* on the
other side. The code will compile — it just may not do what it did
before, so the site is flagged for review.

The classic example is argument passing across the v1/v2 boundary:

```text
warning[W0511]: argument passing changes meaning across the v1/v2 boundary
  note: v1 deep-copies non-`@` arguments and clones return values,
        while v2+ passes containers by reference; a v1 `@` parameter
        aliases even scalars, which v2+ does not — review call sites
        that mutate parameters or returned containers
```

Other flagged sites include returned containers (cloned at v1,
aliased at v2+) and operations whose sharing semantics differ
between versions.

What to do: read the note attached to each warning and check whether
the surrounding code depends on the old behavior — typically whether
anything mutates a value that used to be a private copy and is now
shared (or vice versa). Add explicit `clone(...)` calls where the
old copying behavior must be preserved.

See also W0510 (migration result doesn't compile) and W0512
(construct that cannot be migrated automatically).
